## User Scenarios & Testing *(mandatory)*

### User Story 1 - 品牌识别一致（跨端） (Priority: P1)

企业在 admin（PC）、app（移动）、docs（文档）三端任一处看到「移山 / YISHAN」品牌识别时，**品牌名、Logo 占位、字体、主色** 四个不变项完全一致，不会出现「同一个产品在三个端长得像三个产品」。

**Why this priority**: 品牌识别是企业级产品的第一道门槛。跨端不一致会被用户立刻感知为「不专业」「小作坊」，再多的功能都难以挽回。

**Independent Test**: 同时打开 admin、app、docs 三端，截屏对比以下 4 项：
- 品牌名（应该都是「移山」或「YISHAN」，不应有「移山后台管理系统」「yishan」混用）
- 主色按钮（应该都是 `#1677FF`）
- 主字体（admin 端 AlibabaSans，app 端系统默认）
- Logo 字符标（admin/app 应出现 `⛰`）

**Acceptance Scenarios**:

1. **Given** 打开 yishan-admin 任意页面，**When** 查看 logo 和 title，**Then** 显示「移山」+ `⛰` 字符标，title 不再是「移山后台管理系统」
2. **Given** 打开 yishan-app 登录页或首页，**When** 查看顶部，**Then** 显示「移山」+ `⛰` 字符标，主色按钮使用 `#1677FF`
3. **Given** 打开 yishan-docs 首页，**When** 查看 title 和 brand，**Then** 显示「移山」+ 一致的 brand 色
4. **Given** 任意一端，**When** 点击主操作按钮，**Then** 按钮主色在三个端色值完全相同（用截图取色器验证）
5. **Given** yishan-app 任意文字内容，**When** 渲染，**Then** 使用系统默认字体（无需额外下载，包体更小）

### User Story 2 - 登录页品牌呈现 (Priority: P1)

企业用户首次打开 yishan-app，登录页呈现完整的品牌信息：`⛰` 字符标 + 产品名「移山」+ 品牌主张「简单可依赖的后台基座」，给用户专业、可信、有品牌感的第一印象。

**Why this priority**: 登录页是用户与产品的第一次接触。专业感、可信感在登录页奠定基调。

**Independent Test**: 录屏用户从打开 App 到完成登录的全过程，验证登录页布局、Logo、主张、按钮的视觉表现。

**Acceptance Scenarios**:

1. **Given** 用户打开登录页，**When** 进入页面，**Then** 看到 `⛰` 字符标（48-64px）+ 「移山」产品名（24-28px 主标题）+ 「简单可依赖的后台基座」品牌主张（14px 副标题），布局居中、留白合理
2. **Given** 用户输入账号和密码，**When** 输入框获取焦点，**Then** 输入框边框颜色变化为 `#1677FF`（聚焦态），自动唤起键盘
3. **Given** 用户点击登录，**When** 提交登录，**Then** 登录按钮显示 Loading（不可重复点击），按钮主色 `#1677FF`，成功后平滑跳转到首页

### User Story 3 - 首页看板驾驶舱（大气体感） (Priority: P1)

企业用户（管理员 / 普通用户）打开 yishan-app 首页，立即看到 **顶部主色品牌渐变带 + 大号欢迎语 + 加粗统计卡 + 最近登录** 一气呵成的"企业级"视觉体感。首页在 1 秒内给人"有分量、不寒碜"的第一印象，**不出现"信息少就看着小气"**。

**Why this priority**: 首页是企业 App 的脸面。"大气" = 视觉权重到位 + 留白合理 + 排版讲究 + 主色用得稳。这是用户每天 1-3 次的接触面，气场不到位会持续侵蚀品牌信任。

**Independent Test**: 录屏管理员/普通用户进入首页的完整 1.5 秒，验证：
- 顶部 80-120px 主色品牌渐变带（`⛰` + 「移山」+ 品牌主张）
- 大号欢迎语 `Hi, 姓名`（32-36px 主色，姓名用 600 字重）
- 4 张统计卡数字 32-40px + 卡片圆角 12-16px + 主色淡色背景

**Acceptance Scenarios**:

1. **Given** 任意用户进入首页，**When** 页面加载（≤ 2 秒），**Then** 顶部出现 80-120px 主色渐变品牌带，从主色 `#1677FF` 渐变到 `#0958D9`，带内显示 `⛰`（32px 白色）+「移山」+ 一句品牌主张
2. **Given** 任意用户进入首页，**When** 加载完成，**Then** 欢迎语 `Hi, 姓名` 字号 ≥ 32px，主色 `#1677FF`，姓名 600 字重，下方 13-14px 副标题显示「部门 · 今日日期」
3. **Given** 管理员登录后进入首页，**When** 加载完成，**Then** 看到 4 张统计卡：**用户总数 N**、**部门总数 N**、**今日登录 N**、**当前在线 N**，数字用主色 `#1677FF` 32-40px 加粗显示，卡片圆角 12-16px，背景淡主色（`--color-primary-bg`）
4. **Given** 4 张统计卡，**When** 查看卡片内部，**Then** 卡片包含：顶部 icon 容器（24×24 主色淡色背景）、中部数字（32-40px 主色加粗）、底部标签（12-13px 灰阶文字），三段式信息层次清晰
5. **Given** 4 张统计卡，**When** 用户点击某张卡片，**Then** 跳转到对应管理页（用户卡 → 用户管理；部门卡 → 部门管理；登录卡 → 登录日志；在线卡 → 登录日志）
6. **Given** 普通用户登录后进入首页，**When** 加载完成，**Then** 不显示 4 张统计卡（避免信息冗余），但保留主色品牌渐变带 + 大号欢迎语
7. **Given** 任意用户进入首页，**When** 加载完成，**Then** 整体布局采用 16-20px 区块间距（`--space-4` 到 `--space-5`），不出现"挤在一起"的廉价感
8. **Given** 首页加载完成，**When** 滚动到底部，**Then** 看到「最近登录」列表（最多 5 条，登录时间 + IP + 状态 Tag）
9. **Given** 首页 4 张统计卡的数据，**When** 用户下拉刷新，**Then** 重新拉取聚合接口，刷新数字（带 Loading 指示器）
10. **Given** 聚合接口超时或失败，**When** 加载完成，**Then** 统计卡位置显示「数据加载失败，点击重试」+ 重试按钮，不出现白屏
11. **Given** 普通用户无任何登录记录，**When** 加载完成，**Then** 「最近登录」位置显示「暂无登录记录」空状态

### User Story 4 - 我的页面最核心 3 项 (Priority: P1)

企业用户在「我的」页面看到个人信息 + 3 项核心操作入口（个人资料、修改密码、退出登录），其他功能（系统管理、通讯录、主题等）暂不出现，避免信息过载。

**Why this priority**: 「我的」是企业 App 的基础页面，承载个人信息展示和最关键的系统操作。初版本只做 3 项核心，避免引入未稳定的功能（如系统管理、主题切换）。

**Independent Test**: 进入「我的」页面，验证只出现 3 项菜单 + 退出登录卡片式按钮。

**Acceptance Scenarios**:

1. **Given** 用户进入「我的」页面，**When** 加载完成，**Then** 顶部是个人卡片（头像、姓名、所属部门、岗位），中部是 3 项菜单（个人资料、修改密码、登录日志），底部是「退出登录」按钮（卡片样式，警示色调）
2. **Given** 管理员用户进入「我的」页面，**When** 加载完成，**Then** 顶部个人卡片下方显示「系统管理」入口（仅管理员可见，点击进入系统管理 Tab）
3. **Given** 用户点击「个人资料」，**When** 触发跳转，**Then** 跳转到资料编辑页（PC 端已有等价页），可修改头像、姓名、邮箱、手机号、昵称
4. **Given** 用户点击「修改密码」，**When** 触发跳转，**Then** 进入密码修改页（输入旧密码 + 新密码 + 确认新密码），提交前校验强度和一致性
5. **Given** 用户点击「退出登录」，**When** 触发操作，**Then** 弹出确认 Dialog，确认后清除本地 token、跳转到登录页

### User Story 5 - 应用 Tab 复用（保持现状） (Priority: P2)

「应用」Tab 保持现有 `WorkbenchGrid` 渲染逻辑（从 API 拉菜单树 → 按目录分组渲染），本初版本不做视觉/功能升级。

**Why this priority**: 「应用」Tab 已经是 ToB 内部系统的标准范式（菜单树 + 宫格），本初版本聚焦首页驾驶舱和品牌基线，「应用」Tab 暂不动。

**Independent Test**: 验证「应用」Tab 现有功能不受影响，菜单可正常点击跳转。

**Acceptance Scenarios**:

1. **Given** 用户进入「应用」Tab，**When** 加载完成，**Then** 按菜单目录分组渲染应用宫格，4 列布局（与现状一致）
2. **Given** 用户点击某个应用，**When** 触发跳转，**Then** 正确路由到对应功能页（如用户管理、部门管理等）
3. **Given** 「应用」Tab 加载中 / 空 / 错误，**When** 触发状态，**Then** 显示现有的 `StateView` 反馈（不引入新的三态规范，避免范围爆炸）

### User Story 6 - 底部 TabBar 圆角悬浮胶囊 (Priority: P1)

企业用户在 yishan-app 任意页面，底部 TabBar 呈现**圆角悬浮胶囊形态**：距屏幕底部 8-12px 留白、左右贴边圆角 12-16px、白色背景 + 柔和阴影、激活态 Tab 显示大号 icon + 主色淡色底色 chip，整体"有空气、不贴边"。

**Why this priority**: TabBar 是用户每天触摸 100+ 次的组件。传统"贴底矩形 + 小字"形态显得工具化、低端；圆角悬浮胶囊是企业级 iOS App / 钉钉工作台的"基础配置"。

**Independent Test**: 录屏用户在 3 个 Tab 之间切换，验证：
- TabBar 距屏幕底部 8-12px 留白
- 左右圆角 12-16px
- 激活态有 36×36 主色淡色底色 chip
- 切换时主色高亮、icon 略放大

**Acceptance Scenarios**:

1. **Given** 用户在任意 Tab 页面，**When** 查看底部，**Then** TabBar 距屏幕底部 8-12px，左右贴边圆角 12-16px，背景 `#FFFFFF`，带 `--shadow-md` 阴影
2. **Given** 用户在任意 Tab 页面，**When** 查看激活态 Tab，**Then** 显示 36×36 主色淡色底色 chip（`--color-primary-bg`），icon 颜色 `#1677FF`，文字主色加粗
3. **Given** 用户在任意 Tab 页面，**When** 查看未激活态 Tab，**Then** icon 和文字均为 `--color-text-tertiary`，无底色
4. **Given** 用户点击 Tab 切换页面，**When** 切换完成，**Then** 激活态平滑过渡（背景色 + icon 颜色 transition 200ms），无闪烁
5. **Given** iPhone 等有 Home Indicator 的设备，**When** 查看 TabBar，**Then** 底部安全区适配（增加 12-16px 内边距避免与 Home 条重叠）
6. **Given** 任意 Tab 页面，**When** 滚动内容，**Then** TabBar 固定底部不滚动、不抖动

### User Story 7 - 全局视觉与交互规范（微信小程序 + 标题栏 + 表单 + Icon） (Priority: P1)

yishan-app 的所有页面、子页面、组件、Icon **严格遵循微信小程序官方设计规范**：
- 所有子页面（除 Tab 主页外）使用**系统默认导航栏**（白底黑字 + 返回箭头 + 标题居中），不自定义覆盖
- 所有表单根据字段数自适应布局：≤ 3 字段视口严格垂直居中，> 3 字段顶部对齐
- 所有 Icon 采用**圆角线性风格**（1.5-2px stroke），必要时使用 24×24 主色淡色背景容器
- 跨页面基础元素（按钮、Tag、列表项、输入框）的样式 100% 一致

**Why this priority**: 这 4 条是"企业级 / 微信生态兼容"的基础规范。任何一个不达标都会让用户立刻感知"这不像正经产品"。先把这 4 条定下来，所有后续页面都按这个规范走，避免后期返工。

**Independent Test**: 任意打开 5 个子页面（个人资料/修改密码/系统管理任意一个/任意详情页），验证：
- 标题栏样式统一为白底黑字 + 返回箭头 + 标题居中
- 同一类元素（按钮/Tag/列表项）跨页面像素级一致
- 任意 icon 都能识别为"圆角线性"风格

**Acceptance Scenarios**:

1. **Given** 用户进入任意子页面（不是 Tab 主页），**When** 查看顶部，**Then** 显示微信小程序系统默认导航栏：白底 `#FFFFFF`、黑色文字 `#1D2129`、左侧返回箭头（点击返回上一级）、中间标题居中（与 `app.config.ts` 中 `navigationBarTitleText` 一致）
2. **Given** 用户在子页面内，**When** 滚动内容，**Then** 标题栏固定顶部不滚动、不抖动
3. **Given** 用户进入登录页（字段 ≤ 3），**When** 加载完成，**Then** 登录表单（账号、密码、登录按钮）**视口严格垂直居中**（form 上下边距相等）
4. **Given** 用户进入系统管理「新增用户」页（字段 > 3），**When** 加载完成，**Then** 表单从标题栏下方开始排列（顶部对齐），按字段顺序垂直排布
5. **Given** 任意页面，**When** 查看所有 Icon，**Then** Icon 风格统一为 1.5-2px 圆角线性 stroke（不是细线性、不是实心填充）
6. **Given** 需要区分含义的 Icon（统计卡 icon、TabBar icon 等），**When** 渲染，**Then** 使用 24×24 主色淡色背景容器（`--color-primary-bg`），icon 主色 `#1677FF`
7. **Given** 任意两个页面，**When** 对比同一类元素（按钮/Tag/列表项/输入框），**Then** 视觉外观完全一致（高度、内边距、字号、颜色、圆角）
8. **Given** 任意页面，**When** 查看 spacing（区块间距、组件内边距），**Then** 全部来自 design token（`--space-2` 到 `--space-5`），无裸写 px
9. **Given** 任意页面，**When** 查看配色，**Then** 全部来自 design token（`--color-primary`、`--color-text-*` 等），无裸写 hex

### Edge Cases

- **聚合接口失败**：首页统计卡位置显示「数据加载失败，点击重试」+ 重试按钮；用户点击后重新拉取。
- **Token 失效**：任意 API 调用触发统一拦截，自动跳转到登录页，登录成功后回到原页面（PC 端已有等价行为）。
- **首屏白屏**：冷启动到首页可交互时间 ≤ 2 秒（中端设备），无白屏或长时间 loading。
- **网络异常**：登录页提交前检测无网络，给出明确提示（无网络、请检查网络），按钮可重试。
- **管理员/普通用户切换**：同一账号在不同角色下首页内容自动适配（通过 `useAuthStore.user.roles` 判断）。
- **空数据兜底**：管理员聚合接口返回 0 时，4 张卡片显示 `0`（不是空白/loading），避免「看着像没加载好」。
- **超长用户名/部门名**：在统计卡和欢迎区正确截断或省略号，不撑破布局。

## Requirements *(mandatory)*

### Functional Requirements

#### 品牌基线（Brand Baseline）

- **FR-001**: System MUST 在三端（admin / app / docs）统一展示品牌名为「移山」（YISHAN 为副标识），不再使用「移山后台管理系统」等长尾名称
- **FR-002**: System MUST 在三端统一使用 `⛰` 字符标作为本期 Logo 占位（待正式 Logo 设计后替换）
- **FR-003**: System MUST 在三端统一品牌主色为 `#1677FF`（与 App 现状对齐），admin 端 antd Pro token 同步更新
- **FR-004**: System MUST 在 app 端使用系统默认字体（无需下载字体，包体更小、加载更快；admin 端保持 AlibabaSans）
- **FR-005**: System MUST 在 app 登录页和「我的」页面展示品牌主张「简单可依赖的后台基座」

#### 跨端对齐

- **FR-006**: System MUST 在 admin 端将 antd Pro `colorPrimary` 从 `#1890ff` 改为 `#1677FF`，确保所有主色按钮、链接、Tag 同步切换
- **FR-007**: System MUST 在 admin 端 ProLayout 标题从「移山后台管理系统」改为「移山」，并在 Logo 位置显示 `⛰` 字符标

- **FR-016**: System MUST 在 app 首页实现大号欢迎语 `Hi, 姓名`，字号 ≥ 32px，主色 `#1677FF`，姓名 600 字重，副标题显示部门 + 今日日期

- **FR-022**: System MUST 在 app TabBar 切换时实现平滑过渡（背景色 + icon 颜色 transition 200ms），无闪烁
- **FR-023**: System MUST 在 app TabBar 保持固定底部不滚动、不抖动（与原方案一致的固定行为）

#### 我的页面

- **FR-024**: System MUST 在 app「我的」页面仅展示 3 项核心菜单（个人资料、修改密码、登录日志）+ 底部「退出登录」卡片按钮

- **FR-028**: System MUST 保持「应用」Tab 现有的 `StateView` 反馈组件（Loading / Error / Empty），不引入新的三态规范

#### 全局视觉与交互规范

- **FR-029**: System MUST 严格遵循微信小程序官方设计规范（安全区、导航栏、TabBar、按钮、列表、Icon）
- **FR-030**: System MUST 在所有子页面（除 Tab 主页外）使用微信小程序系统默认导航栏（白底 `#FFFFFF`、黑色文字 `#1D2129`、左侧返回箭头、中间标题居中），不在 Taro 中自定义 navigationStyle
- **FR-031**: System MUST 在所有轻表单（≤ 3 字段）场景中实现视口严格垂直居中（form 上下边距相等），适用于登录页、重置密码等
- **FR-032**: System MUST 在所有复杂表单（> 3 字段）场景中实现顶部对齐（从标题栏下方开始排列），适用于新增用户、编辑资料等

- **FR-034**: System MUST 在需要区分含义的 Icon 场景（统计卡、TabBar、菜单项）中使用 24×24 主色淡色背景容器（`--color-primary-bg`），icon stroke 颜色 `#1677FF`
- **FR-035**: System MUST 保证跨页面基础元素（按钮、Tag、列表项、输入框、Icon）的样式 100% 一致，所有样式值（颜色、字体、间距、圆角、阴影）必须引用 design token，禁止裸写 hex/px/shadow 字符串

### Key Entities *(include if feature involves data)*

- **BrandBaseline（品牌基线）**: 跨端共享的品牌常量，定义品牌名、Logo 占位、主色、字体、主张；初版本在代码中硬编码，后续可抽到 `packages/brand`
- **DashboardStats（首页看板数据）**: 管理员首页聚合数据，4 个数字字段：
  - `userTotal: number` — 用户总数
  - `deptTotal: number` — 部门总数
  - `todayLogin: number` — 今日登录次数
  - `online: number` — 当前在线人数
- **UserRole（用户角色）**: 通过 `useAuthStore.user.roles` 判断当前用户是管理员还是普通用户，决定首页内容

## Assumptions

- 目标用户为已注册的企业内部用户（管理员、普通员工），他们有稳定的网络（4G/WiFi），但偶有弱网和断网场景
- yishan-app 同时支持微信小程序和 H5 端，本初版本聚焦核心功能，三端差异限于平台限制
- yishan-app 已经在 `src/styles/tokens.scss` 中定义了基础 design token，本次升级将基于现有 token 体系进一步完善
- 后端 API 能力已基本就绪（用户、部门、字典、登录日志等接口），本初版本新增 1 个聚合接口 `/api/v1/app/dashboard/stats`，其他接口保持不变
- PC 端 admin 已有「个人资料」「修改密码」「登录日志」功能，本初版本在 app 端做等价入口
- 管理员/普通用户的角色判断通过 `useAuthStore.user.roles` 完成（admin 端已有等价逻辑）

- 性能指标（中端设备、首屏 ≤ 2 秒）基于主流移动设备（iPhone 12+ / Android 中端骁龙 7 系+）作为参考
- 用户类型假设：企业内部用户，预期每日使用 1-3 次，每次 3-10 分钟，关注效率而非娱乐性
