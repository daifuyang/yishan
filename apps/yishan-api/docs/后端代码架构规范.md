# 后端代码架构与模块开发规范

> 适用范围：`apps/yishan-api` 的新模块、重构模块和代码审查。  
> 目标：任何开发者或自动化工具都能按同一结构交付可读、可测、可演进的业务代码。

## 1. 架构目标与不可违背的规则

Yishan API 采用**模块化单体**：模块内部高内聚，模块之间通过稳定的 Service 契约协作；未来可按模块边界拆分为独立服务。

必须遵守以下规则：

1. 路由层不直接访问数据库，不写业务规则。
2. Service 是业务用例和事务边界；跨 Repository 的操作必须由 Service 编排。
3. Repository 只处理一个聚合的数据访问和关联同步，不依赖 Fastify、HTTP 请求/响应、缓存或其他业务 Service。
4. Schema 是外部 API 契约的唯一来源；数据库实体、内部输入对象和 API DTO 不可混用。
5. 所有写操作必须可追踪：操作者、更新时间、软删除状态和必要的并发策略必须明确。
6. 依赖方向只能向内：`Route → Service → Repository → DB`。内层不得反向依赖外层。

```text
HTTP Request
    │
    ▼
Route ── 参数校验、鉴权、HTTP 响应
    │
    ▼
Service ── 用例、业务规则、事务、缓存协调、DTO 映射
    │
    ▼
Repository ── 查询、持久化、关联同步
    │
    ▼
Drizzle / MySQL
```

## 2. 模块目录与归属

新业务模块应放在 `src/plugins/modules/<module>/`，并保持模块内实现完整：

```text
src/plugins/modules/<module>/
├── routes/api/v1/
│   ├── index.ts                 # 公开接口
│   └── admin/
│       ├── autohooks.ts         # 管理域统一鉴权
│       └── <resource>/index.ts  # 资源路由
├── schemas/                     # TypeBox 请求/响应契约
├── services/                    # 用例编排与业务规则
├── repositories/                # 数据访问
├── mappers/                     # API DTO / 持久化输入转换（复杂模块必须独立）
├── constants/                   # 模块业务常量、枚举
├── exceptions/                  # 模块业务异常（需要时）
└── test/                        # 模块就近测试
```

历史核心模块仍位于 `src/core/` 时，保持相同层次：

```text
src/core/
├── routes/
├── schemas/
├── services/
├── repositories/
├── mappers/
└── test/
```

禁止通过 `export * from '@/legacy/...'` 为新模块做空壳转发；模块必须拥有自己的实现、测试和契约。

## 3. 各层职责

### 3.1 Route：只处理 HTTP

Route 只能做以下工作：

- 声明 Fastify route、TypeBox Schema、`operationId`、`tags`、`security`；
- 取得 `request.params/query/body/currentUser`；
- 调用一个 Service 用例；
- 使用 `ResponseUtil` 返回统一响应；
- 处理 HTTP 级保护，例如“不能删除自己”“超级管理员不可禁用”。

Route 禁止：

- 导入 `drizzleDb`、表对象或编写 SQL；
- 处理密码哈希、角色关联、唯一性校验等领域逻辑；
- 在多个 endpoint 复制业务判断；
- 返回未经过 Schema 定义的内部字段。

```ts
fastify.post('/', { schema: createUserRouteSchema }, async (request, reply) => {
  const user = await UserService.create(request.body, request.currentUser.id, fastify);
  return ResponseUtil.success(reply, user);
});
```

### 3.2 Service：一个方法对应一个业务用例

Service 是业务规则的唯一入口。一个方法应描述一个可读的用例，如 `create`、`update`、`disable`、`delete`、`resetPassword`，而不是暴露 SQL 风格的方法。

Service 必须负责：

- 将 API Request DTO 转为内部持久化输入；
- 密码哈希、格式规范化、默认值和业务校验；
- 唯一性、权限、状态迁移等跨记录规则；
- 跨 Repository 的事务；
- 缓存读取、刷新、失效与领域操作的协调；
- 将实体/读模型映射为 API Response DTO；
- 将数据库唯一键冲突等基础设施错误转换为业务错误。

Service 禁止：

- 把 `FastifyRequest`、`FastifyReply` 传给 Repository；
- 拼接 SQL 或直接操作表；
- 为了“方便”把请求 DTO 原样 spread 给数据库；
- 在一个方法中混入无关用例，例如“更新用户”同时发通知、生成报表、执行批量迁移。

复杂转换应提取 Mapper：

```ts
export class UserService {
  static async create(request: CreateUserReq, operatorId: number, fastify?: FastifyInstance) {
    await this.ensureUniqueIdentity(request);

    const input = UserMapper.toCreateInput({
      request,
      operatorId,
      passwordHash: await hashPassword(request.password),
    });
    const user = await UserRepository.create(input);
    await this.refreshCache(user, fastify);
    return UserMapper.toResponse(user);
  }
}
```

### 3.3 Repository：只处理数据访问

新模块必须使用 `XxxRepository` 命名，并放在 `repositories/` 目录。`*Model` 是历史兼容命名：仅允许维护既有代码，禁止在新模块或新文件中继续使用；重构一个历史模块时应将其一并迁移为 Repository。

Repository 可以做：

- 选择列、构建受控查询条件、排序、分页；
- 插入、更新、软删除与查询受影响行；
- 同一聚合内的关联同步，例如用户与角色、用户与部门；
- 接收 `AppQueryDb`，以便被 Service 的事务复用；
- 返回实体、只读投影或内部结果对象。

Repository 禁止：

- 接收 `FastifyRequest`、`FastifyReply` 或 HTTP 状态码；
- 接收 `CreateXxxReq` / `UpdateXxxReq` 等 HTTP DTO；
- 密码哈希、缓存操作、发送消息、鉴权决策；
- 直接调用其他业务的 Service；
- 使用 `any` 绕过实体、输入或查询结果类型。

内部输入必须显式定义，避免 API 字段直接落库：

```ts
export interface UpdateUserInput {
  updaterId: number;
  nickname?: string;
  status?: number;
  birthDate?: Date | null; // undefined: 不更新；null: 显式清空
  passwordHash?: string;
  deptIds?: number[];
  roleIds?: number[];
}

static async update(id: number, input: UpdateUserInput, db: AppQueryDb = drizzleDb) {
  const { deptIds, roleIds, ...fields } = input;
  await db.update(sysUser).set({ ...fields, updatedAt: dateUtils.now() })
    .where(and(eq(sysUser.id, id), isNull(sysUser.deletedAt)));
  // 仅同步本聚合的关联。
}
```

### 3.4 Mapper：边界转换的唯一位置

当字段转换超过少量、存在枚举显示名、日期格式或多个调用端时，必须创建 Mapper。

Mapper 负责：

- `Request DTO → Service/Repository Input`；
- `Entity / Read Projection → Response DTO`；
- 数字枚举与公开字符串枚举的转换；
- `Date ↔ ISO 字符串`、空字符串与 `null` 的语义转换。

Mapper 不访问数据库、不做网络调用、不抛业务权限错误。

## 4. 数据与事务规范

### 4.1 事务边界

单表简单写入可由 Repository 单独完成；只要一次用例涉及两个以上持久化动作，就由 Service 开启事务并传递 `tx`：

```ts
await drizzleDb.transaction(async (tx) => {
  await UserRepository.softDelete(id, tx);
  await UserTokenRepository.revokeAllByUserId(id, tx);
  await AuditLogRepository.create(auditInput, tx);
});
```

事务中禁止调用外部 HTTP、发邮件、发布不可回滚消息或执行长耗时任务。此类副作用应在提交后触发，或通过 outbox/event 表异步处理。

### 4.2 写入语义

- `undefined`：调用方未提供字段，禁止覆盖原值；
- `null`：明确表示清空可空字段；
- 空字符串是否允许必须由 Schema 定义，并在 Mapper 中转换；
- 创建和更新都必须明确设置 `creatorId`、`updaterId`、`createdAt`、`updatedAt` 的归属；
- 软删除必须始终附带 `deletedAt is null` 条件；
- 批量关联输入必须 `Set` 去重；空数组表示清空关联。

### 4.3 并发与唯一性

- 数据库唯一索引是最终事实；Service 的预检查仅用于返回友好错误；
- 写入时必须将唯一冲突转换成确定的业务错误；
- 对存在并发覆盖风险的资源，Schema 提供 `version`，更新使用 `id + version` 条件并检查 affected rows；
- 不需要乐观锁时，不保留“看似存在但从未参与条件”的 `version` 字段。

### 4.4 查询、分页与排序

- 对外排序字段必须在 Schema 中枚举，并映射到受控 Drizzle 列引用；
- 不得把 `sortBy`、列名、表名或 update field 从请求字符串直接带入 ORM；
- 列表查询的筛选条件、排序和分页参数只规范化一次，列表和总数复用同一过滤条件；
- `page >= 1`、`1 <= pageSize <= 100`，并确保响应分页元数据使用规范化后的值；
- 查询只选择业务需要的列；非认证流程不得读取 `passwordHash`、token 等敏感字段。

## 5. 契约、异常与安全

### 5.1 Contract-first

修改 API 的顺序固定为：

1. 修改 TypeBox Request/Response Schema；
2. 修改 Route 的 OpenAPI 声明；
3. 实现 Service、Repository、Mapper；
4. 补路由和 Service/Repository 测试；
5. 运行 `pnpm --filter yishan-admin openapi`（对外契约变化时）；
6. 更新相关文档。

所有公开字段为 lower camel case；MySQL 使用 snake case；Drizzle 属性使用 lower camel case。跨边界必须显式转换。

### 5.2 异常

- 参数格式问题交由 TypeBox 校验；
- 业务规则失败使用 `BusinessError` 和稳定业务码；
- Repository 不知道 HTTP 状态码和展示文案；
- 不将原始数据库错误、堆栈、密码、token、连接串返回给客户端或写入普通日志。

### 5.3 安全基线

- 密码只在 Service 哈希，Repository 只接收 `passwordHash`；
- 响应 DTO 必须默认排除密码、refresh token、API token hash、软删除和内部版本字段；
- 权限校验在 Route hook 和 Service 业务规则两层明确表达；
- 删除、禁用、改密、授权、导出等高风险操作必须记录审计信息；
- 缓存 key 必须包含资源类型和 ID；写入成功后刷新或失效相应缓存。

## 6. 测试规范

| 层级 | 必测内容 | 允许 Mock |
| --- | --- | --- |
| Route | Schema 校验、鉴权、HTTP 状态、统一响应、关键权限限制 | Service |
| Service | 业务规则、DTO 转换、事务编排、缓存协调、错误转换 | Repository、缓存客户端 |
| Repository | 查询条件、软删除、关联同步、排序白名单、分页边界 | 不应 Mock Drizzle 查询本身；使用测试库或受控 DB |

每个写用例至少覆盖：

- 成功路径；
- 参数缺失/非法；
- 权限或状态限制；
- 唯一冲突或目标不存在；
- 部分更新不覆盖未传字段；
- 事务中的任一步失败会整体回滚。

每个列表接口至少覆盖：默认排序、每个公开排序字段、分页边界、筛选条件与总数一致性。

## 7. 命名与代码风格

- 文件：`kebab-case.ts`；类：`PascalCase`；方法/变量：`camelCase`；常量：`UPPER_SNAKE_CASE`；
- 方法使用业务动词：`create`、`findById`、`list`、`softDelete`、`revokeAll`；避免 `handleData`、`process`、`doSomething`；
- `find*` 返回实体或 `null`，`list*` 返回数组，`count*` 返回数字；
- 不使用 `any`。未知输入用 `unknown` 并在边界收窄；
- 不用非空断言 `!` 掩盖 Schema 与类型不一致；
- 一个函数超过约 40 行、出现两层以上业务分支、或承担两个职责时，应拆出私有函数/Mapper/用例；
- 注释说明“为什么”和不变量，不复述代码行为。

## 8. 新模块交付清单

在创建模块或提交重构前逐项确认：

- [ ] 模块目录、路由前缀、Schema、Service、Repository 和测试已创建；
- [ ] Route 不直接访问 DB；Repository 不依赖 HTTP/缓存/其他 Service；
- [ ] 所有公开输入均由 TypeBox Schema 约束；
- [ ] 请求 DTO、内部输入、持久化实体、响应 DTO 的转换边界明确；
- [ ] 动态查询使用枚举/白名单映射；
- [ ] 写操作具备事务、审计字段、软删除和并发策略；
- [ ] 敏感字段不在普通查询和响应中出现；
- [ ] Route、Service、Repository 三层测试覆盖到位；
- [ ] `build:ts`、相关测试、OpenAPI 生成（如适用）均通过；
- [ ] 契约或规范变化已更新文档。

## 9. 现有代码迁移方式

历史模块不要求一次重写。每次触及某模块时，按以下顺序渐进迁移：

1. 先补充或修正 Schema 与测试；
2. 将历史 `*Model` 迁移为 `*Repository`，并移出 HTTP DTO、密码哈希、缓存和跨聚合事务；
3. 引入显式内部输入、Mapper 与安全投影；
4. 收敛动态查询和分页逻辑；
5. 清除无调用方的旧方法，避免新旧接口长期并存。

新模块必须从第一天完整遵守本规范并使用 Repository 命名；已有模块按“修改即改善”原则迁移。
