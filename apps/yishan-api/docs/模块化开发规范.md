# 模块化开发规范（AutoLoad）

本文档定义 `src/plugins/modules` 下模块开发、路由组织、鉴权与测试规范。目标是让后续 CLI 生成模块时，目录、路径与行为保持一致。

## 1. 模块加载机制

应用入口通过 AutoLoad 自动加载 `src/plugins/modules`，并启用目录级 hooks 级联：

- `autoHooks: true`
- `cascadeHooks: true`

这意味着：

- 模块目录中的 `autohooks.ts` 会自动生效
- 子目录路由会继承上层 `autohooks.ts` 的 hook
- 模块可以通过目录结构自然表达“公开路由/鉴权路由”

## 2. 目录约定

以 `hello` 模块为基准：

```text
src/plugins/modules/hello/
├── routes/
│   └── api/
│       └── v1/
│           ├── index.ts         # 版本级公开路由（如 /ping、/echo）
│           └── admin/
│               ├── autohooks.ts # 管理域统一鉴权 hook
│               ├── index.ts     # 管理域根路由（如 /admin）
│               └── me/
│                   └── index.ts # 管理域子路由（如 /admin/me）
└── test/
    └── routes.test.ts           # 模块就近测试
```

路径规则：

- 目录名决定路由片段，如 `routes/api/v1/admin/me/index.ts` => `/api/v1/admin/me`
- `index.ts` 表示当前目录根路由
- 不在文件内重复硬编码上层前缀

推荐模块目录（与 `src` 对齐）：

```text
src/plugins/modules/<module>/
├── routes/              # 模块路由
├── services/            # 模块服务（完整实现，避免仅 re-export 根目录）
├── models/              # 模块数据访问
├── schemas/             # 模块 schema
├── constants/           # 模块常量
├── exceptions/          # 模块异常
├── utils/               # 模块工具
└── test/                # 模块测试
```

## 3. 路由规范

- 使用 `FastifyPluginAsync`
- 每个端点都声明 `schema.summary/description/operationId/tags`
- 鉴权端点声明 `security: [{ bearerAuth: [] }]`
- 返回统一使用 `ResponseUtil.success`
- 参数校验使用 TypeBox（`Type.Object` / `$ref`）

## 4. 鉴权规范

管理域统一使用目录级 `autohooks.ts`：

```ts
fastify.addHook('preHandler', async (request, reply) => {
  return fastify.authenticate(request, reply)
})
```

约束：

- 在 `admin/`（及其子目录）内，不重复添加 `preHandler: fastify.authenticate`
- 公开路由放在无 `autohooks.ts` 影响的目录

## 5. 测试规范

- 模块测试就近放在 `test/` 目录
- 文件命名：`*.test.ts`
- 测试路径和模块实际前缀保持一致
- 覆盖最少三类场景：
  - 公开接口成功
  - 鉴权成功
  - 鉴权失败（401）

Vitest 已配置扫描：

- `test/**/*.test.ts`
- `src/**/test/**/*.test.ts`

## 6. CLI 生成约定（建议）

后续 CLI 新建模块时最少生成：

- `src/plugins/modules/<module>/routes/api/v1/index.ts`
- `src/plugins/modules/<module>/routes/api/v1/admin/autohooks.ts`
- `src/plugins/modules/<module>/test/routes.test.ts`

## 7. 测试用例指南

模块测试用例按“路由契约”组织，建议至少覆盖以下三类：

- 公开接口成功：验证状态码、`success` 与核心返回字段
- 鉴权成功：携带有效 `Authorization`，验证业务字段与权限路径
- 鉴权失败：缺少或错误 token，验证状态码（如 401）与错误结构

测试编写要点：

- 测试文件固定放在 `test/*.test.ts`
- `buildApp` 中模拟 `authenticate` 与 `currentUser`
- 测试 URL 与模块实际前缀保持一致
- 断言优先覆盖状态码、统一响应结构、关键业务字段

参考示例：

- `src/plugins/modules/hello/test/routes.test.ts`

## 8. Hello 模块基准

当前 `hello` 模块已作为基准实现，满足以下要求：

- AutoLoad 目录化路由
- 公开与鉴权路径分层
- 目录级鉴权 hook
- 模块就近测试

建议后续模块直接对齐 `hello` 目录结构再扩展业务资源。

## 9. 架构收益与微服务演进

当前采用模块化 AutoLoad 的单体架构，核心收益如下：

- 模块边界清晰：按目录划分公开域、管理域、资源域，职责稳定
- 研发效率更高：CLI 可按固定结构快速生成模块并直接接入
- 变更影响可控：模块内部迭代不会影响其他模块的路由组织
- 运维路径统一：日志、鉴权、错误处理通过插件与 hook 统一治理

该结构天然支持后续拆分为微服务，建议按“模块边界”进行渐进式拆分：

- 第一步：保持单体部署，先完成模块内业务闭环
- 第二步：按模块抽离服务契约（DTO、错误码、鉴权策略）
- 第三步：将高负载或高独立性的模块单独部署为服务
- 第四步：通过网关或 BFF 聚合，保持外部 API 稳定

## 10. 插件扩展规范
- 模块支持插件化扩展，推荐在 `src/plugins/modules/<module>/plugins/` 下组织扩展能力。

推荐插件类型：

- 领域插件：如缓存策略、模块内审计、领域事件分发
- 接口插件：如模块限流、模块级请求幂等、路由前置校验
- 集成插件：如第三方通知、外部系统对接适配器

插件约束：

- 插件只处理横切能力，不直接承载核心业务流程
- 插件通过 `fastify.decorate` 或 hook 暴露能力，避免全局污染
- 插件命名应体现模块归属，避免与全局插件重名
- 插件依赖关系应在模块内显式声明，避免隐式耦合

模块迁移约束：

- 模块目录中的 `services/models/schemas/constants/exceptions/utils` 需提供完整实现
- 禁止通过 `export * from '@/xxx'` 方式做单纯转发
- 目标是模块可直接复制到其他项目并快速落地
