# SysUserModel 关键测试场景

本文档记录了 SysUserModel 应覆盖的核心测试场景。虽然当前使用模拟测试，但建议未来使用真实测试数据库进行集成测试。

## 1. 用户生日部分更新
- **场景**：更新用户时，只修改昵称或其他字段，不提供 birthDate 字段
- **预期结果**：生日字段应保持不变，不应被设置为 null
- **验证点**：
  - 检查更新 SQL 语句中是否仅包含 birthDate 当且仅当该字段在请求中提供
  - 验证更新后的用户数据中 birthday 保持原样

## 2. 分页边界测试
- **场景 1**：page 为 0，pageSize 为 10
  - 预期：page 自动调整为 1，返回第 1 页结果
- **场景 2**：pageSize 为 1000
  - 预期：pageSize 自动调整为 100
- **场景 3**：pageSize 为 0
  - 预期：pageSize 自动调整为 1
- **验证点**：
  - 验证 Service 层返回的 pagination 元数据与实际查询一致
  - 验证数据库查询使用的 limit 和 offset 正确

## 3. 重复关联 ID 去重
- **场景**：创建或更新用户时，deptIds 或 roleIds 包含重复项
  - 例如：deptIds: [1, 1, 2, 2, 3]
- **预期结果**：
  - 保存到数据库的关联数据应只有去重后的结果
  - [1, 2, 3]
- **验证点**：
  - 检查 sysUserDept 和 sysUserRole 表中是否没有重复记录
  - 验证返回的用户详情中 deptIds 和 roleIds 已去重

## 4. 删除事务一致性
- **场景 1**：正常删除用户
  - 预期：
    - 用户表中 deletedAt 被设置
    - 用户的部门关联 deletedAt 被设置
    - 用户的角色关联 deletedAt 被设置
    - 用户的所有令牌被撤销
    - 所有操作在同一事务中，要么全部成功，要么全部失败

- **场景 2**：删除过程中发生错误
  - 预期：整个事务回滚，数据恢复到删除前状态
  - 验证：
    - 用户未被删除
    - 关联关系未被删除
    - 令牌未被撤销

## 5. 唯一性校验查询优化
- **场景**：检查用户名/邮箱/手机号唯一性
- **预期**：
  - 数据库查询仅返回 id 字段，不返回其他字段，特别是不返回 passwordHash
  - 使用新添加的 getUserIdByUsername / getUserIdByEmail / getUserIdByPhone 方法
- **验证**：
  - 检查 SQL 查询的 SELECT 子句是否只包含 id 列
  - 验证这些方法的返回类型为 number | null

## 6. 更新时间字段
- **场景**：创建、更新、删除用户时
- **预期**：
  - 创建时：createdAt 和 updatedAt 设置为当前时间
  - 更新时：updatedAt 设置为当前时间
  - 删除时：updatedAt 设置为当前时间
- **验证**：
  - 检查数据库中时间字段是否被正确设置
  - 检查返回的响应中时间字段是否包含正确的 ISO 格式

## 设置集成测试环境

若要实现真实的集成测试，建议：

1. **创建测试数据库配置**
   - 在 apps/yishan-api/.env.test 中配置测试数据库
   - 使用单独的测试数据库，避免污染开发环境

2. **测试前准备**
   - 每次测试前重置数据库状态
   - 使用 Drizzle 的 migrate 功能创建测试表
   - 可选：使用种子数据填充测试数据

3. **测试后清理**
   - 每个测试后回滚事务或清理测试数据

4. **测试隔离**
   - 每个测试独立运行，不依赖其他测试的状态
