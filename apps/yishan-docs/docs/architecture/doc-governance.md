# 文档治理

文档站 `apps/yishan-docs/docs` 是团队对外和对内的主文档入口。根目录、`apps/yishan-api/docs`、`apps/yishan-admin/docs` 下的 Markdown 可以保留为源码附近说明、历史设计或深度规范，但应避免与文档站重复维护同一事实。

## 分层原则

- 入门文档：帮助新成员在最短路径内跑起来。
- 架构文档：解释系统边界、核心流程、插件体系和关键取舍。
- 开发文档：说明前端、后端、OpenAPI、权限、规范与常用命令。
- 模块文档：描述业务模块的页面、API、权限、数据模型和边界规则。
- 运维文档：描述本地开发、CI/CD、FC3 部署、环境变量和排障流程。

## 单篇文档建议结构

模块类文档建议包含：

```text
# 模块名称

## 适用范围
## 页面入口
## API 与服务
## 权限点
## 数据模型
## 关键业务规则
## 常见问题
```

架构类文档建议包含：

```text
# 主题

## 背景
## 当前实现
## 设计约束
## 影响范围
## 后续演进
```

## 维护要求

- 涉及路径、命令、路由、环境变量、部署流程的变更，应同步更新文档站。
- 明确区分“当前实现”和“规划中能力”。规划内容应标注状态，避免误导使用者。
- 避免写入个人机器路径、固定测试账号、密钥、AccessKey、数据库连接串等敏感或不可复用信息。
- PR 中如果修改 API、菜单、部署、插件 manifest，应检查对应文档是否需要更新。
